# 产品想法：A股全量信息检索知识库

## 问题陈述
A股市场信息高度分散，财报、公告、政策、研报、社交媒体热点等数据源割裂，个人投资者和量化研究者难以在单一平台完成"采集→清洗→检索→分析"的闭环。现有工具要么侧重行情数据（Wind、同花顺iFinD），要么侧重资讯聚合（东方财富），缺乏一个以"事件链+因子"为核心、支持毫秒级检索和ML训练的统一知识库。本项目通过三层数据架构（Bronze → Silver → Gold）和自动化采集流水线，解决这一信息碎片化问题。

## 产品愿景
构建一个A股全量信息检索知识库，采用三层架构（Bronze → Silver → Gold），实现24小时自动搜集A股板块及5000+股票相关信息，支持毫秒级查询和ML训练。

## 核心架构
- **Bronze层**：不可变原始冻结数据（一次写入，永不修改）
- **Silver层**：清洗统一后的结构化数据
- **Gold层**：多维因子 + ML特征 + 归档聚合

## 核心功能
1. **增量采集**：财报、资金、公告、政策、研报、合同签订
2. **热点追踪**：雪球网、东方财富网、同花顺、淘股吧等
3. **数据处理**：哈希去重 → 倒排索引 → 个股事件链
4. **智能分析**：LLM消化 → 情感因子 → 推荐引擎
5. **查询能力**：毫秒级SQL查询（单票/条件筛选/事件画像）
6. **策略支持**：因子IC评估 + ML训练 + 组合回测

## 核心原则
- 一次采集多处复用
- 统一写入分散消费
- 当天数据24h内冻结入Bronze不可变层

## 目标用户与Job-to-be-Done
- **主要用户**：个人量化投资者 / 独立研究员（1人使用场景）
- **核心任务**：当我关注某只股票或板块时，我需要快速获取其历史事件全貌、市场情绪和相关研报，以便制定或验证交易策略
- **次要用户**：AI Agent（通过API消费数据用于策略生成和内容创作）

## 使用场景
1. **驾驶舱通知栏**：滚动输出最新热点及高价值线索
2. **推荐引擎**：运行时多维认证
3. **内容生成**：研报晨报公众号文章素材
4. **个股画像**：快速全面生成个股详细分析
5. **ML训练**：策略进化数据源

## 成功标准
- 覆盖5000+ A股股票
- 24小时内完成数据冻结
- 查询响应时间 < 100ms
- 数据去重准确率 > 99%
- 支持至少5个并发使用场景

## 约束

- 个人项目，无团队支持，MVP 阶段仅支持核心链路
- 数据采集需遵守目标网站的 robots.txt 和使用条款
- 数据源为公开信息，不涉及付费数据库（Wind/Bloomberg）

## 非目标

- 不做交易执行或经纪业务
- 不构建面向大众的SaaS产品（个人工具定位）
- 不替代专业量化平台（如JoinQuant、RiceQuant），而是提供数据层

# CONTEXT — A股全量信息检索知识库

## 术语表（Glossary）

| 术语 | 定义 |
|------|------|
| **Bronze层** | 数据湖三层架构中的原始层，存储不可变的原始采集数据。一旦写入即冻结，用于审计和重放。 |
| **Silver层** | 清洗后的结构化数据层，完成去重、格式统一、实体对齐，是下游消费的主要数据源。 |
| **Gold层** | 聚合/特征层，包含多维因子计算结果、ML训练特征集和归档摘要，直接服务于分析和策略。 |
| **个股事件链** | 按时间顺序串联某只股票相关的所有事件（公告、财报、资金异动、舆情），形成可追溯的因果链。 |
| **因子IC（Information Coefficient）** | 因子值与未来收益率的秩相关系数，用于评估因子的预测能力。IC > 0.03 通常认为因子有效。 |
| **T+1数据** | 交易日结束后（次日）才可用的数据，区别于盘中实时数据。本项目的数据新鲜度目标。 |
| **倒排索引** | 从关键词到文档列表的映射结构，Elasticsearch 的核心检索机制，支持全文搜索。 |
| **哈希去重** | 对采集内容计算哈希值（如SHA-256），相同哈希视为重复内容，确保数据唯一性。 |
| **ETL管道** | Extract-Transform-Load 数据流水线，负责从 Bronze→Silver→Gold 的逐层转换。 |

## 领域上下文

- **A股市场**：中国大陆沪深两市，约5000+上市公司
- **数据源类型**：财报/公告（官方披露）、资金流（交易数据）、政策（政府发布）、研报（机构产出）、舆情（社交媒体）
- **用户画像**：个人量化投资者，具备Python/SQL基础，有策略回测需求
- **部署环境**：单台VPS，Docker Compose编排

## 关键实体

- **股票**：以股票代码（如 600519.SH）为唯一标识
- **板块**：行业/概念分类，一只股票可归属多个板块
- **事件**：发生在某只股票或板块上的离散信息单元
- **因子**：从原始数据中计算得出的数值特征，用于ML和选股

## ADR-001：采用三层数据架构（Bronze/Silver/Gold）

**状态**：已采纳
**日期**：2026-06-12
**决策者**：pm-aligner（基于brief推导）

### 上下文

- ✅ 数据重放和审计成为可能
- ✅ 各层可独立扩展和替换
- ⚠️ 增加数据管道复杂度，需维护ETL逻辑

## ADR-002：数据存储采用 PostgreSQL + Elasticsearch 组合

**状态**：已采纳
**日期**：2026-06-12
**决策者**：pm-aligner（基于brief推导）

### 上下文
需要同时支持结构化查询（SQL筛选/关联）和全文检索（公告/研报搜索）。

### 决策
- PostgreSQL 存储结构化数据（股票元数据、因子值、事件链关系）
- Elasticsearch 存储全文内容（公告原文、研报、舆情），提供倒排索引

### 后果
- ✅ 各自发挥所长，SQL和全文检索性能最优
- ⚠️ 需维护两套数据库，增加运维复杂度
- ⚠️ 单VPS上同时运行 PG + ES 对内存压力大（建议≥8GB）
- ⚠️ 数据一致性需通过应用层保证（非分布式事务）

## 2. 技术栈调研

### 2.1 数据采集层

#### Scrapy + Playwright 组合方案

金融数据采集面临的核心挑战：
- **反爬策略**：目标网站（雪球、东方财富、同花顺）均有反爬机制
- **动态渲染**：大量数据通过JS异步加载，静态HTML无法获取
- **规模化稳定性**：单机脚本难以稳定运行，需重试、调度、IP轮换

推荐技术栈：
- **Scrapy**：负责请求分发、调度、管道化数据处理
- **Playwright**：负责JS渲染页面抓取、动态交互
- **Redis**：任务队列（生产者/消费者模型），多机扩展
- **代理IP池**：应对限速和封禁

架构演进路径：单机验证 → 并发多标签 → Redis分布式任务队列

| 组件 | 最低内存 | 推荐内存 | 说明 |
|------|---------|---------|------|
| PostgreSQL | 1GB | 2~4GB | 结构化数据存储，事件链关联查询 |
| Elasticsearch | 2GB | 4~8GB | 全文检索倒排索引，堆内存≤50%系统RAM，上限31GB |
| Redis | 512MB | 1~2GB | 缓存+任务队列 |
| 采集进程 | 512MB | 1GB | Scrapy+Playwright运行时 |
| FastAPI后端 | 256MB | 512MB | API服务 |
| **合计** | **~4.5GB** | **~9GB** | 不含系统开销 |

**关键发现**：
- Elasticsearch 官方推荐最小 4 vCPU + 32GB RAM 用于生产集群，但在单节点小型部署中，1~2GB堆内存可支撑5000+股票的全文检索
- Elasticsearch 全文搜索延迟可达 sub-1ms 级别（在合理分片和索引设计下）
- PostgreSQL 与 ES 在同一VPS上运行可行，但需严格控制 ES 堆内存（不超过系统RAM的50%）
- 对于5000+股票、T+1数据量的场景，建议至少 **8GB RAM** 的VPS

### 2.3 AI/LLM 分析层

#### LLM在金融量化中的应用

**情感分析（Sentiment Analysis）**：
- LLM（GPT-4、Llama2-7B等）在金融新闻标题情感分析上超越FinBERT和传统BERT方法
- 社交媒体（StockTwits、雪球、淘股吧）情感分析可预测短期股价走势
- 关键优势：处理口语化表达、表情符号、行业术语、讽刺语境

**因子挖掘（Alpha Mining）**：
- **Alpha-GPT**：人机交互驱动的因子生成，用户自然语言描述→LLM生成因子表达式
- **Few-Shot学习**：在高质量因子示例基础上模仿生成，超越传统遗传算法
- **知识引导提示（SKGP）**：将交易想法转换为领域特定提示，提高因子质量
- 每轮生成的因子约40%语法语义正确，需迭代修正

**LLM成本估算**：
- GPT-3.5 API：$0.5~2/1M tokens
- 本地部署开源模型（Qwen、Llama3等）：零API成本，但需要GPU资源
- 月度token成本：假设每日处理5000条资讯×200 tokens/条 = 1M tokens/天 ≈ ¥10~30/天（GPT-3.5），月度 ¥300~900
- 使用本地部署开源模型可降至接近零成本

## 3. 用户调研与需求验证

### 3.1 目标用户画像验证

根据brief中定义的目标用户（个人量化投资者/独立研究员），市场调研验证：

- **个人投资者对专业终端的需求有限**：雪球社区讨论显示，多数个人投资者认为"投资核心是理念提升而非工具"，关注几家公司即可，不需要覆盖全市场
- **中小型私募是"中间地带"**：需要专业数据深度但对价格敏感，是AI工具红利的直接受益者
- **迁移成本高**：更换终端面临工作流和历史数据迁移壁垒，这是Wind护城河的核心

### 3.2 差异化定位机会

1. **事件链检索**：Wind/iFinD提供资讯列表，但不按"个股事件链"组织。本项目以时间线串联事件（公告→财报→资金异动→舆情），是独特差异化点。
2. **三层数据架构**：Bronze（原始冻结）→ Silver（清洗对齐）→ Gold（因子/特征），这种分层在个人工具中几乎无竞品。
3. **零订阅费**：对比Wind 3.98万/年、Choice 1.8万/年，个人自建方案的成本优势显著。
4. **ML训练数据源**：Wind/Choice不提供结构化因子数据集用于ML训练，本项目的Gold层直接服务此需求。

## 4. 风险评估

### 4.1 高风险项

| 风险 | 影响 | 缓解策略 | 置信度 |
|------|------|----------|--------|
| **VPS资源不足** | PostgreSQL+ES同时运行可能OOM | MVP阶段使用SQLite替代ES做验证；或用Meilisearch（更轻量）替代 | MEDIUM |
| **反爬封锁** | 雪球/东方财富等限制爬虫频率 | 使用AKShare/Tushare等合法API替代直接爬虫；遵守robots.txt | HIGH |
| **LLM成本超标** | 每日大量情感分析导致API费用过高 | 优先本地部署开源模型（Qwen2.5-7B等）；批量处理而非实时 | MEDIUM |
| **数据源不稳定** | 免费数据接口可能变更或下线 | 多数据源冗余（AKShare+Tushare+自建爬虫）；Bronze层冻结保证数据不丢失 | HIGH |

### 4.2 中风险项

| 风险 | 影响 | 缓解策略 | 置信度 |
|------|------|----------|--------|
| **ES查询延迟>100ms** | 违反成功标准 | 合理分片设计、预聚合、缓存热查询 | MEDIUM |
| **数据去重<99%** | 重复数据影响分析质量 | SHA-256哈希去重+相似度匹配（文本内容） | HIGH |
| **5000+股票覆盖不全** | 部分板块/个股数据缺失 | 分阶段覆盖：先沪深300→中证500→全市场 | HIGH |

## 5. 推荐技术路线（MVP）

```
Bronze层（不可变原始数据）
├── AKShare API → 股票行情/财报数据 → JSON存储
├── Tushare Pro → 财务指标/宏观数据 → JSON存储
└── Scrapy+Playwright → 雪球/淘股吧舆情 → HTML快照+文本

Silver层（清洗结构化）
├── 哈希去重（SHA-256）
├── 实体对齐（股票代码标准化）
├── 事件链构建（按股票+时间排序）
└── PostgreSQL（结构化存储）

Gold层（因子/特征）
├── LLM情感因子（本地Qwen2.5-7B）
├── Elasticsearch（全文检索+倒排索引）
└── Redis（热查询缓存）

查询层
├── FastAPI REST接口
└── 毫秒级全文检索（ES）+ 条件筛选（PostgreSQL）
```

## 6. 结论与建议

### 6.1 竞品格局总结

本项目在A股金融数据市场中定位为**个人投资者的AI增强型知识库**，与Wind/iFinD/Choice等机构终端形成互补而非竞争关系。核心差异化在于：
1. 事件链+因子导向（而非传统行情/资讯导向）
2. 三层数据架构（Bronze/Silver/Gold）
3. 零订阅费的个人工具定位
4. 直接服务ML训练的数据输出

### 6.2 关键决策建议

1. **优先使用AKShare/Tushare等合法API**，减少爬虫依赖，降低法律风险
2. **VPS建议升级至8GB+ RAM**，以支持PostgreSQL+ES+Redis同时运行（当前假设4核8GB需验证）
3. **LLM优先本地部署**开源模型（Qwen2.5-7B/14B），控制API成本在¥500/月以内
4. **MVP阶段聚焦单股票事件链**完整链路，而非全量5000+股票，验证技术可行性后再扩展

### 6.3 开放假设验证结果

| 假设编号 | 原假设 | 验证结论 | 调整建议 |
|----------|--------|----------|----------|
| 1 | VPS 4核8GB足够 | **需升级** | ES+PG同时运行建议≥8GB RAM，16GB更佳 |
| 2 | 公开页面可稳定爬取 | **高风险** | 优先用AKShare/Tushare API，爬虫仅作补充 |
| 3 | ES <100ms响应 | **可行** | 单索引+合理分片设计下可达到sub-100ms |
| 4 | LLM成本<¥500/月 | **可行** | 本地部署开源模型可接近零成本 |
| 5 | 用户接受T+1 | **已确认** | 非目标中已明确 |
| 6 | PG+ES同VPS可运行 | **可行** | 需严格控制内存分配 |
