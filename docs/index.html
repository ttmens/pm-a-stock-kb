# 产品想法：A股全量信息检索知识库

## 问题陈述
A股市场信息高度分散，财报、公告、政策、研报、社交媒体热点等数据源割裂，个人投资者和量化研究者难以在单一平台完成"采集→清洗→检索→分析"的闭环。现有工具要么侧重行情数据（Wind、同花顺iFinD），要么侧重资讯聚合（东方财富），缺乏一个以"事件链+因子"为核心、支持毫秒级检索和ML训练的统一知识库。本项目通过三层数据架构（Bronze → Silver → Gold）和自动化采集流水线，解决这一信息碎片化问题。

## 产品愿景
构建一个A股全量信息检索知识库，采用三层架构（Bronze → Silver → Gold），实现24小时自动搜集A股板块及5000+股票相关信息，支持毫秒级查询和ML训练。

## 核心架构
- **Bronze层**：不可变原始冻结数据（一次写入，永不修改）
- **Silver层**：清洗统一后的结构化数据
- **Gold层**：多维因子 + ML特征 + 归档聚合

## 核心功能
1. **增量采集**：财报、资金、公告、政策、研报、合同签订
2. **热点追踪**：雪球网、东方财富网、同花顺、淘股吧等
3. **数据处理**：哈希去重 → 倒排索引 → 个股事件链
4. **智能分析**：LLM消化 → 情感因子 → 推荐引擎
5. **查询能力**：毫秒级SQL查询（单票/条件筛选/事件画像）
6. **策略支持**：因子IC评估 + ML训练 + 组合回测

## 核心原则
- 一次采集多处复用
- 统一写入分散消费
- 当天数据24h内冻结入Bronze不可变层

## 目标用户与Job-to-be-Done
- **主要用户**：个人量化投资者 / 独立研究员（1人使用场景）
- **核心任务**：当我关注某只股票或板块时，我需要快速获取其历史事件全貌、市场情绪和相关研报，以便制定或验证交易策略
- **次要用户**：AI Agent（通过API消费数据用于策略生成和内容创作）

## 使用场景
1. **驾驶舱通知栏**：滚动输出最新热点及高价值线索
2. **推荐引擎**：运行时多维认证
3. **内容生成**：研报晨报公众号文章素材
4. **个股画像**：快速全面生成个股详细分析
5. **ML训练**：策略进化数据源

## 成功标准
- 覆盖5000+ A股股票
- 24小时内完成数据冻结
- 查询响应时间 < 100ms
- 数据去重准确率 > 99%

## 术语表（Glossary）

| 术语 | 定义 |
|------|------|
| **Bronze层** | 数据湖三层架构中的原始层，存储不可变的原始采集数据。一旦写入即冻结，用于审计和重放。 |
| **Silver层** | 清洗后的结构化数据层，完成去重、格式统一、实体对齐，是下游消费的主要数据源。 |
| **Gold层** | 聚合/特征层，包含多维因子计算结果、ML训练特征集和归档摘要，直接服务于分析和策略。 |
| **个股事件链** | 按时间顺序串联某只股票相关的所有事件（公告、财报、资金异动、舆情），形成可追溯的因果链。 |
| **因子IC（Information Coefficient）** | 因子值与未来收益率的秩相关系数，用于评估因子的预测能力。IC > 0.03 通常认为因子有效。 |
| **T+1数据** | 交易日结束后（次日）才可用的数据，区别于盘中实时数据。本项目的数据新鲜度目标。 |
| **倒排索引** | 从关键词到文档列表的映射结构，Elasticsearch 的核心检索机制，支持全文搜索。 |
| **哈希去重** | 对采集内容计算哈希值（如SHA-256），相同哈希视为重复内容，确保数据唯一性。 |
| **ETL管道** | Extract-Transform-Load 数据流水线，负责从 Bronze→Silver→Gold 的逐层转换。 |

## 领域上下文

- **A股市场**：中国大陆沪深两市，约5000+上市公司
- **数据源类型**：财报/公告（官方披露）、资金流（交易数据）、政策（政府发布）、研报（机构产出）、舆情（社交媒体）
- **用户画像**：个人量化投资者，具备Python/SQL基础，有策略回测需求
- **部署环境**：单台VPS，Docker Compose编排

## 关键实体

- **股票**：以股票代码（如 600519.SH）为唯一标识
- **板块**：行业/概念分类，一只股票可归属多个板块
- **事件**：发生在某只股票或板块上的离散信息单元
- **因子**：从原始数据中计算得出的数值特征，用于ML和选股

### 2.1 数据采集层

#### Scrapy + Playwright 组合方案

金融数据采集面临的核心挑战：
- **反爬策略**：目标网站（雪球、东方财富、同花顺）均有反爬机制
- **动态渲染**：大量数据通过JS异步加载，静态HTML无法获取
- **规模化稳定性**：单机脚本难以稳定运行，需重试、调度、IP轮换

推荐技术栈：
- **Scrapy**：负责请求分发、调度、管道化数据处理
- **Playwright**：负责JS渲染页面抓取、动态交互
- **Redis**：任务队列（生产者/消费者模型），多机扩展
- **代理IP池**：应对限速和封禁

架构演进路径：单机验证 → 并发多标签 → Redis分布式任务队列

### 2.3 AI/LLM 分析层

#### LLM在金融量化中的应用

**情感分析（Sentiment Analysis）**：
- LLM（GPT-4、Llama2-7B等）在金融新闻标题情感分析上超越FinBERT和传统BERT方法
- 社交媒体（StockTwits、雪球、淘股吧）情感分析可预测短期股价走势
- 关键优势：处理口语化表达、表情符号、行业术语、讽刺语境

**因子挖掘（Alpha Mining）**：
- **Alpha-GPT**：人机交互驱动的因子生成，用户自然语言描述→LLM生成因子表达式
- **Few-Shot学习**：在高质量因子示例基础上模仿生成，超越传统遗传算法
- **知识引导提示（SKGP）**：将交易想法转换为领域特定提示，提高因子质量
- 每轮生成的因子约40%语法语义正确，需迭代修正

**LLM成本估算**：
- GPT-3.5 API：$0.5~2/1M tokens
- 本地部署开源模型（Qwen、Llama3等）：零API成本，但需要GPU资源
- 月度token成本：假设每日处理5000条资讯×200 tokens/条 = 1M tokens/天 ≈ ¥10~30/天（GPT-3.5），月度 ¥300~900
- 使用本地部署开源模型可降至接近零成本

### 5.2 常用查询

#### 股票查询

```sql
-- 查询所有股票
SELECT stock_code, stock_name, industry FROM stocks ORDER BY stock_code;

-- 按行业筛选
SELECT stock_code, stock_name FROM stocks WHERE industry = '白酒';

-- 模糊搜索股票名称
SELECT stock_code, stock_name FROM stocks WHERE stock_name LIKE '%茅台%';

-- 查询股票详情
SELECT * FROM stocks WHERE stock_code = '600519.SH';
```

#### 事件查询

```sql
-- 查询某股票的事件链 (按时间倒序)
SELECT e.event_id, e.event_type, e.event_time, e.title, e.source, e.sentiment_score, s.stock_name
FROM events e
JOIN stocks s ON e.stock_code = s.stock_code
WHERE e.stock_code = '600519.SH'
ORDER BY e.event_time DESC;

-- 查询某股票近 N 天的事件
SELECT * FROM events
WHERE stock_code = '600519.SH'
  AND event_time >= datetime('now', '-30 days')
ORDER BY event_time DESC;

-- 按事件类型筛选
SELECT * FROM events
WHERE stock_code = '600519.SH' AND event_type = 'announcement'
ORDER BY event_time DESC;

-- 查询情感评分最高的事件 (TOP 10)
SELECT stock_code, title, sentiment_score FROM events
ORDER BY sentiment_score DESC LIMIT 10;

-- 按事件类型统计
SELECT event_type, COUNT(*) as cnt FROM events GROUP BY event_type ORDER BY cnt DESC;

-- 按来源统计
SELECT source, COUNT(*) as cnt FROM events GROUP BY source ORDER BY cnt DESC;
```

#### 全文搜索 (FTS5)

```sql
-- 全文搜索关键词
SELECT e.event_id, e.title, e.event_type, e.stock_code, e.sentiment_score,
       highlight(events_fts, 0, '[', ']') as highlighted_title,
       highlight(events_fts, 1, '[', ']') as highlighted_content
FROM events_fts
WHERE events_fts MATCH '芯片'
ORDER BY rank;

-- 多关键词搜索 (AND)
SELECT * FROM events_fts WHERE events_fts MATCH '芯片 AND 制裁';

-- 按股票代码过滤的全文搜索
SELECT e.* FROM events e
INNER JOIN events_fts f ON e.event_id = f.rowid
WHERE f MATCH '茅台' AND e.stock_code = '600519.SH';

-- 按事件类型过滤的全文搜索
SELECT e.* FROM events e
INNER JOIN events_fts f ON e.event_id = f.rowid
WHERE f MATCH '业绩' AND e.event_type = 'financial';

-- 重建 FTS 索引 (数据不同步时)
INSERT INTO events_fts(events_fts) VALUES('rebuild');
```

#### 因子数据查询

```sql
-- 查询某股票的因子数据
SELECT factor_date, factor_name, factor_value FROM factor_values
WHERE stock_code = '600519.SH'
ORDER BY factor_date DESC;
```

## 1. 北极星指标（North Star Metric）

### 业务类型判定：Productivity Game（效率游戏）

本产品是**个人量化投资者的研究效率工具**。用户价值不在于停留时长或交易次数，而在于**多快地完成个股研究并导出可用数据**。竞品 Wind/iFinD/Choice 是 SaaS 订阅模式，本项目是零订阅费、单 VPS 自部署的生产力工具。

### 北极星指标

> **周活跃研究会话数（Weekly Active Research Sessions, WARS）**
>
> 定义：单个部署实例在一周内产生的"有效研究会话"总数。一个有效研究会话 = 用户执行了至少一次事件链查询、全文搜索或因子数据导出操作（三次操作之一即计为一个会话，同一用户同一次连续操作计为一次会话）。

### 为什么选择 WARS（七项标准验证）

| 标准 | 验证 |
|------|------|
| 易于理解 | "一周内用户做了多少次研究查询"——团队每个人都懂 |
| 以客户为中心 | 衡量用户实际获得的研究价值，而非部署数量或服务器收入 |
| 可持续价值 | 周复一周的活跃会话反映习惯养成和长期留存 |
| 愿景对齐 | 产品愿景是"为个人量化投资者提供零订阅费的 A 股知识库"——用户每周都在用它做研究，就是愿景兑现 |
| 可量化 | 通过 FastAPI 日志统计查询次数，按周聚合 |
| 可行动 | 数据覆盖率、搜索速度、部署体验都直接影响会话数 |
| 领先指标 | 周活跃会话增长预示用户留存和口碑传播，先于任何变现信号 |

### 输入指标（Input Metrics）

| # | 输入指标 | 定义 | 影响 NSM 的路径 | 当前基线 | 90 天目标 |

# Proposal: A股全量信息检索知识库 — 自研轻量平台

## Why

个人量化投资者和独立研究员在A股市场分析中面临信息碎片化与工具成本高昂的双重困境。Wind/iFinD/Choice 等机构终端年费 1.4万~10万+，个人投资者难以承受；AKShare/Tushare 等开源工具仅解决数据采集层，缺乏检索引擎、知识库架构和LLM分析能力。

市场上不存在**以"个股事件链"为核心组织方式、零订阅费、单VPS部署**的A股知识库产品。本项目填补这一空白，为个人量化投资者提供：

- 按时间线串联的个股事件链（公告→财报→资金异动→舆情）
- 毫秒级全文检索（公告/研报/舆情）
- LLM情感因子生成，直接服务ML训练
- 完全数据自主，无第三方SaaS依赖

## What changes

### 新增能力

1. **Bronze→Silver→Gold 三层数据管道**
   - Bronze层：不可变原始数据冻结（JSON/HTML）
   - Silver层：清洗、去重、实体对齐、事件链构建（PostgreSQL）
   - Gold层：LLM情感因子、全文检索索引（Elasticsearch）

2. **个股事件链检索**
   - 按股票代码查询时间线事件序列
   - 支持事件类型过滤（公告/财报/资金/舆情）
   - 因果追溯：关联事件的时间窗口匹配

3. **全文检索引擎**
   - 公告/研报/舆情内容全文搜索
   - 关键词高亮、分页、按股票/时间过滤
   - P95查询延迟 <100ms

4. **LLM情感分析**
   - 本地部署Qwen2.5-7B（GGUF 4-bit量化）
   - 批量离线处理新闻标题、舆情文本
   - 输出情感因子（-1~+1）存入Gold层

5. **REST API + 简单Web界面**
   - 事件链查询、全文搜索、因子导出接口
   - 单用户token认证（轻量级）

### 技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 全文检索 | Elasticsearch 8.x | sub-100ms延迟，成熟生态 |
| 结构化存储 | PostgreSQL 15 | 事件链关联查询，SQL表达能力 |
| 缓存/队列 | Redis 7 | 热查询缓存，ETL任务调度 |
| API框架 | FastAPI + Uvicorn | 异步高性能，OpenAPI自动生成 |
| LLM推理 | Ollama / llama.cpp | 本地部署，零API成本 |
| 部署方式 | Docker Compose | 单VPS编排，简化运维 |
| 数据源 | AKShare + Tushare + 爬虫 | 合法API优先，爬虫补充舆情 |

## Impact

### 对用户的影响

- **零订阅费**：替代年费1.4万~10万+的机构终端
- **数据自主**：所有数据存储在个人VPS，无第三方依赖
- **ML就绪**：Gold层直接输出因子数据集，服务量化策略研发
- **事件链视角**：不同于传统终端的资讯列表，按时间线组织事件

### 对技术架构的影响

- 新增9个容器（见 `architecture/c4-container.md`）
- 新增7个API组件（见 `architecture/c4-component.md`）
- 需要8GB+ RAM的VPS环境
- 建立T+1批量采集和ETL调度机制

### 风险和缓解

| 风险 | 缓解策略 |
|------|----------|
| VPS内存不足 | MVP阶段SQLite原型验证；ES堆内存限制2GB |
| 数据源API限制 | 多数据源冗余；遵守频率限制；Bronze层冻结 |
| LLM推理速度慢 | 批量离线处理；量化模型；CPU优化 |
| 单点故障 | 每日备份到对象存储；Bronze层支持快速重建 |

## 数据模型设计

### Silver层 (PostgreSQL)

```sql
-- 股票元数据表
CREATE TABLE stocks (
    stock_code VARCHAR(20) PRIMARY KEY,  -- 如 600519.SH
    stock_name VARCHAR(100) NOT NULL,
    industry VARCHAR(50),
    list_date DATE,
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 事件表 (事件链核心)
CREATE TABLE events (
    event_id BIGSERIAL PRIMARY KEY,
    stock_code VARCHAR(20) REFERENCES stocks(stock_code),
    event_type VARCHAR(30) NOT NULL,  -- announcement/financial/capital/social
    event_time TIMESTAMP NOT NULL,
    title VARCHAR(500),
    content_hash VARCHAR(64) UNIQUE NOT NULL,  -- SHA-256去重
    source VARCHAR(100),
    raw_data_id VARCHAR(200),  -- Bronze层文件路径
    created_at TIMESTAMP DEFAULT NOW()
);

-- 因子值表 (Gold层输出)
CREATE TABLE factor_values (
    factor_id BIGSERIAL PRIMARY KEY,
    stock_code VARCHAR(20) REFERENCES stocks(stock_code),
    factor_date DATE NOT NULL,
    factor_name VARCHAR(50) NOT NULL,  -- sentiment/momentum/volatility
    factor_value DECIMAL(10,6),
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(stock_code, factor_date, factor_name)
);

-- 索引
CREATE INDEX idx_events_stock_time ON events(stock_code, event_time DESC);
CREATE INDEX idx_events_type ON events(event_type);
CREATE INDEX idx_factors_stock_date ON factor_values(stock_code, factor_date);
```

## 核心流程设计

### 1. T+1 数据采集流程

```
定时触发 (每日18:00)
  → 数据采集器启动
  → 并行拉取：
    ├─ AKShare: 沪深300成分股行情/财报
    ├─ Tushare: 财务指标/宏观数据
    └─ 爬虫: 雪球/淘股吧舆情
  → 写入Bronze层 (JSON/HTML冻结)
  → 记录采集日志 (采集数量/失败项)
  → 触发ETL管道
```

### 2. ETL 转换流程

```
Bronze层新数据就绪
  → SHA-256哈希去重 (跳过已存在content_hash)
  → 实体对齐 (股票代码标准化: 600519 → 600519.SH)
  → 事件链构建 (按stock_code+event_time排序)
  → 写入PostgreSQL (Silver层)
  → 写入Elasticsearch (Gold层全文索引)
  → 触发LLM情感分析任务
```

### 3. LLM 情感分析流程

```
ETL完成后触发
  → 从Silver层获取待分析文本 (title + content摘要)
  → 批量发送至本地LLM服务 (每批50条)
  → LLM返回情感因子 (-1~+1)
  → 写入factor_values表 (factor_name='sentiment')
  → 更新ES索引 (添加sentiment_score字段)
```

### 4. 用户查询流程

```
用户请求 → API网关
  → 路由分发
  ├─ /api/events/{code} → 事件链查询引擎 → PostgreSQL
  │   → 缓存检查 (Redis) → 命中则返回
  │   → 未命中 → SQL查询 → 写入缓存 → 返回
  ├─ /api/search → 全文检索引擎 → Elasticsearch
  │   → 关键词分词 → 倒排索引查询 → 高亮处理 → 返回
  └─ /api/factors/{code} → 因子导出服务 → PostgreSQL
      → 查询factor_values → Pandas处理 → CSV/JSON返回
```

## User Story

US-1：作为个人投资者，我想按时间线查看某只股票的完整事件链，以便快速了解该股票的近期动态

## Requirements

### REQ-1.1 股票搜索与匹配
- 用户输入股票代码（6位数字）或股票名称，系统返回匹配的股票列表
- 匹配优先级：精确代码匹配 > 名称前缀匹配 > 名称模糊匹配
- 返回结果≤10条，包含：stock_code, stock_name, industry
- 响应时间：<3秒

### REQ-1.2 事件链查询
- 输入：stock_code + 时间范围（7天/30天/90天/自定义）
- 输出：按 event_time DESC 排序的事件列表
- 每条事件包含：event_id, event_type, event_time, title, sentiment_score, source
- 支持的事件类型：announcement（公告）、financial（财报）、capital（资金）、social（舆情）

### REQ-1.3 事件类型过滤
- 用户可点击类型标签过滤事件链
- 过滤后即时刷新，无需重新请求全量数据
- 前端缓存全量事件链，过滤为客户端操作

### REQ-1.4 事件详情展示
- 点击事件条目弹出详情面板
- 详情包含：完整内容、来源链接、发布时间、情感评分、关联事件
- 关联事件：同一股票前后5天内的其他事件（最多5条）

### REQ-1.5 情感因子可视化
- 事件条目右侧显示情感因子色块
- 正面（>0.2）：绿色；负面（<-0.2）：红色；中性：灰色
- 色块显示情感因子数值和简短标签（利好/利空/中性）

- 导出格式：CSV
- CSV列名：stock_code, factor_date, factor_name, factor_value

# A股全量信息检索知识库 — MVP

## 概述

基于三层架构（Bronze → Silver → Gold）的A股全量信息检索知识库，支持24小时自动采集、毫秒级查询、ML训练数据导出。

- **后端**: Python 3.11 + FastAPI
- **数据库**: PostgreSQL (结构化) + Elasticsearch (全文检索)
- **缓存**: Redis
- **前端**: 静态HTML + Vanilla JS
- **部署**: Docker Compose

### 5.2 常用查询

#### 股票查询

```sql
-- 查询所有股票
SELECT stock_code, stock_name, industry FROM stocks ORDER BY stock_code;

SELECT stock_code, stock_name FROM stocks WHERE industry = '白酒';

-- 模糊搜索股票名称

```

#### 事件查询

```sql
-- 查询某股票的事件链 (按时间倒序)
SELECT e.event_id, e.event_type, e.event_time, e.title, e.source, e.sentiment_score, s.stock_name
FROM events e
JOIN stocks s ON e.stock_code = s.stock_code
WHERE e.stock_code = '600519.SH'
ORDER BY e.event_time DESC;

-- 查询某股票近 N 天的事件
SELECT * FROM events
WHERE stock_code = '600519.SH'
  AND event_time >= datetime('now', '-30 days')
ORDER BY event_time DESC;

-- 按事件类型筛选
SELECT * FROM events
WHERE stock_code = '600519.SH' AND event_type = 'announcement'
ORDER BY event_time DESC;

-- 查询情感评分最高的事件 (TOP 10)
SELECT stock_code, title, sentiment_score FROM events
ORDER BY sentiment_score DESC LIMIT 10;

-- 按事件类型统计
SELECT event_type, COUNT(*) as cnt FROM events GROUP BY event_type ORDER BY cnt DESC;

-- 按来源统计
SELECT source, COUNT(*) as cnt FROM events GROUP BY source ORDER BY cnt DESC;
```

#### 全文搜索 (FTS5)

```sql
-- 全文搜索关键词
SELECT e.event_id, e.title, e.event_type, e.stock_code, e.sentiment_score,
       highlight(events_fts, 0, '[', ']') as highlighted_title,
       highlight(events_fts, 1, '[', ']') as highlighted_content
FROM events_fts
WHERE events_fts MATCH '芯片'
ORDER BY rank;

-- 多关键词搜索 (AND)
SELECT * FROM events_fts WHERE events_fts MATCH '芯片 AND 制裁';

-- 按股票代码过滤的全文搜索
SELECT e.* FROM events e
INNER JOIN events_fts f ON e.event_id = f.rowid
WHERE f MATCH '茅台' AND e.stock_code = '600519.SH';

-- 按事件类型过滤的全文搜索
SELECT e.* FROM events e
INNER JOIN events_fts f ON e.event_id = f.rowid
WHERE f MATCH '业绩' AND e.event_type = 'financial';

-- 重建 FTS 索引 (数据不同步时)
INSERT INTO events_fts(events_fts) VALUES('rebuild');
```

#### 因子数据查询

```sql
-- 查询某股票的因子数据
SELECT factor_date, factor_name, factor_value FROM factor_values
WHERE stock_code = '600519.SH'
ORDER BY factor_date DESC;

-- 查询特定因子 (如 sentiment)
SELECT stock_code, factor_date, factor_value FROM factor_values
WHERE factor_name = 'sentiment' AND stock_code = '600519.SH'
ORDER BY factor_date DESC;

-- 查询最新因子数据
SELECT stock_code, factor_name, factor_value, factor_date FROM factor_values f1
WHERE factor_date = (SELECT MAX(factor_date) FROM factor_values f2 WHERE f2.stock_code = f1.stock_code AND f2.factor_name = f1.factor_name)
ORDER BY stock_code, factor_name;

-- 因子趋势分析 (某股票 sentiment 趋势)
SELECT factor_date, factor_value FROM factor_values
WHERE stock_code = '600519.SH' AND factor_name = 'sentiment'
ORDER BY factor_date;

-- 全市场因子统计
SELECT factor_name,
       AVG(factor_value) as avg_val,
       MIN(factor_value) as min_val,
       MAX(factor_value) as max_val,
       COUNT(*) as cnt
FROM factor_values
GROUP BY factor_name;
```

## 7. 风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| **部署门槛过高**：目标用户有 Python/SQL 基础，但 Docker + ES + PG + LLM 的部署复杂度可能劝退 | 部署成功率低 → 零 WARS | E6 优先优化；提供预配置 VPS 镜像或一键脚本 |
| **数据合规风险**：AKShare/Tushare 等数据源的许可条款可能限制商业使用 | 项目可能面临法律风险 | README 明确"个人研究用途"；仅使用合法 API |
| **社区反响冷淡**：量化投资者可能不信任自部署方案的数据质量 | 增长实验全部失败 | 先在小范围私募群内测，获取种子用户反馈后再公开发布 |
| **VPS 成本感知**：虽然零订阅费，但 VPS 月费（¥100-200/月）可能成为障碍 | 个人用户放弃部署 | 在文章中对比"VPS 月费 ¥150 vs Wind 年费 ¥3.98 万"，凸显性价比 |
